# 设计原则与注意事项

## 核心原则

### 1. 真实性

只从用户的实际发言中提取模式,不要臆测或添加不存在的内容。

**要做的:**
- 基于用户的实际发言
- 保持原始语境
- 如实记录用户的观点

**不要做的:**
- 推测用户可能的想法
- 添加未表达的观点
- 美化或修饰用户的表达

### 2. 可追溯性

每个提取的要素都要注明来源(哪次会议,哪段发言)。

**实现方式:**
- 在每个要素后标注 `**来源**: [会议名称/日期]`
- 保存原始会议记录到 `meeting-records/`
- 保存提取历史到 `extraction-history/`

**好处:**
- 用户可以验证提取的准确性
- 可以追溯决策模型的演变
- 便于更新或修正错误

### 3. 客观性

在应用模型时,保持分析的客观性,不要过度模仿用户的语气,而是用用户的思维框架来分析。

**正确做法:**
- 使用用户的思考框架
- 应用用户的判断标准
- 参考用户的风险偏好
- 保持专业的分析语气

**错误做法:**
- 完全模仿用户的说话方式
- 使用用户的口头禅
- 过度个性化的表达

### 4. 隐私保护

会议记录可能包含敏感信息,所有文件都保存在本地 `~/Eleon/` 目录。

**安全措施:**
- 所有数据仅存储在本地
- 不上传到云端或第三方服务
- 文件权限设置为用户私有

**敏感信息处理:**
- 如果会议记录包含敏感信息,提醒用户
- 在提取时可以选择性忽略敏感内容
- 保持原始记录的完整性

### 5. 持续演进

决策模型应该随着新会议记录的加入而不断完善,但要保持核心的一致性。

**更新策略:**
- 新模式与现有模式一致 → 强化现有模式
- 新模式与现有模式冲突 → 记录变化,询问用户
- 新模式补充现有模式 → 扩展模型

**保持一致性:**
- 核心价值观应该相对稳定
- 思考框架可以逐步完善
- 具体案例可以持续积累

### 6. 明确说明

在应用模型分析问题时,始终明确告知用户这是基于他们的决策模型,并解释原因。

**透明度要求:**
- 明确说明使用了决策模型
- 指出使用了模型中的哪些要素
- 解释为什么这样分析符合用户的思维方式
- 标注分析的来源(哪次会议/哪个原则)

**示例表述:**
- "根据你的决策模型..."
- "这符合你在[某次会议]中提到的[某个原则]"
- "基于你的风险偏好..."
- "参考你常用的[某个思维模型]"
